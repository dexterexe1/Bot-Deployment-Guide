## What works now

- Landing page at `/` (index.tsx)
- Login page at `/login` (Discord OAuth2)
- Auth callback at `/auth/callback`
- Protected app shell at `/_app` (requires Discord login session)
- Developer portal at `/_app/developer-portal` — has LIVE cursor controls that PATCH /api/v1/site-settings
- CustomCursor component fetched from MongoDB via GET /api/v1/site-settings in __root.tsx
- Bot pushes status to /api/bot/status (authenticated with BOT_API_SECRET header)
- All MongoDB models: User, BotStatus, BotConfig, SiteSettings, GuildResourceSnapshot, etc.

## Key architecture rules

- Dashboard uses its own `src/lib/api.ts` (NOT @workspace/api-client-react) — all API calls go through `apiRequest()`
- API base is `/api/v1` (proxied to port 8080 in dev via vite.config.ts proxy)
- Custom cursor: GET /api/v1/site-settings (public) + PATCH /api/v1/site-settings (requireAuth)
- Dashboard tsconfig has NO reference to api-client-react (removed — dashboard doesn't use it)
- TanStack Router v1 file-based routing; routeTree.gen.ts is auto-generated by vite plugin on dev start
- Session cookie name: `ubdash.sid`; stored in MongoDB via connect-mongo

## Required env vars

- SESSION_SECRET (≥16 chars, set in Replit Secrets)
- MONGO_URI (MongoDB Atlas connection string)
- DISCORD_CLIENT_ID, DISCORD_CLIENT_SECRET, DISCORD_REDIRECT_URI
- BOT_TOKEN (bot only)
- DASHBOARD_APP_URL (e.g. https://your-dashboard.onrender.com)
- WEB_ORIGIN (dashboard origin for CORS, e.g. https://your-dashboard.onrender.com)
- BOT_API_SECRET (shared secret between bot.py and API server)

## Cursor control flow

1. `__root.tsx` fetches GET /api/v1/site-settings on every page load (staleTime 60s)
2. Passes `cursor.enabled`, `cursor.type`, `cursor.color` to `<CustomCursor />`
3. Dev portal `CursorControlCard` does useQuery + useMutation for live editing
4. PATCH /api/v1/site-settings requires a valid session (requireAuth middleware)
5. After save, queryClient.invalidateQueries(['site-settings']) refreshes the cursor everywhere

**Why:** This is the only place the Mongo-backed SiteSettings model is consumed on the frontend.
